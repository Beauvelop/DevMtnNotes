    So let's say I've got this box
    

    
    position: static;
    left: 50vw;
    

    

    left attr is useless without position: relative; in order to make it moveable.

    There. That's better.
    position: relative;
    left: 50vw;
We could even put a top: 10vw; in there to offset it from it's home. 
    position: relative;
    left: 50vw;
    top: 5vw;
It will not disturb 'the flow' of other stuff around it. Remember: It is anchored to, or truly "occupies" the space way to the left and above.
Ok, so in this next static box, let's nest a span in the html:
    < div class="myBox">
        < span>Hi World!< /span>
    < /div>
    I'm right here.
    .myBox {
        width: 40vw;
        height: 40vw;
        position: static;
    }

Let's change the span's position to absolute 



    .myBox {
        position: static;
    }
    span {
        right: 0px;
        position: absolute;
    }
I'm over here! 

    The browser renders element absolutely...at a position 0px from the right.

    However, as soon as its parent is positioned "relative".

    

    .myBox {
        position: relative;
    }
    span {
        right: 0px;
        position: absolute;
    }

    Parent gives child a relative context, so we get:
*Phew! I'm back. 

    The child is positioned absolutely...relative to it's parent.

    Parent gives it 'relative' context within which to be absolute.

    Another example:

    < div class="mybox">
        < div id="toprt">in the corner!
        < / div>
    < / div>
In the corner! 
    .mybox {
        position: relative;
    }
    #toprt {
        top: 0px;
        right: 0px;
        position: absolute;
    }
Z-Index FUN FACT!

    Z-Index vis-a-vis positioning is often overlooked.
    

    Z-Index ONLY "engages" under 3 condtions:

    The Z-indexed element itself MUST be either:
    

    1. fixed
    2. relative
    3. absolute
    

    
This is true regardless of the parent's positioning!
